# コンセプト

## このプロジェクトは何か

markdown で書いたシナリオを、古風な2D RPG として遊べるようにするプロジェクト。
シナリオは AI で「育てる」ことを前提に設計する。

## 決定事項（初期）

| 論点 | 決定 | 理由 |
|---|---|---|
| 技術スタック | **Godot 4** | 2D RPG の王道。Win/Mac/Linux/Android/iOS/Web へネイティブ書き出し。タイルマップ等が充実 |
| 言語/版 | **通常版・GDScript** | Web含むクロスプラットフォームが素直（C#のWeb書き出しは実験的）。AI生成・反復が速く、2D RPGに性能も十分。重い処理はエンジン外の `tools/` 側 |
| AIの関わり方 | **開発時に育てる** | ゲーム本体はAI非依存で動かす。配布・再現性・コストで有利。実行時AIは将来の実験対象 |
| 最初のスコープ | **1本のゲーム先行** | 小さなRPGを1本動かし、共通部分を後からエンジンとして抽出する |
| データの源 | **markdown が真実の源** | 人もAIも読み書きしやすい。バージョン管理・差分レビューに強い |
| マップの源 | **markdown内のASCII** | ビジュアルエディタを源にするとAIによる育成・差分レビューが困難になるため、ASCIIを源に保ち補助ツールで支える |
| レンダラ | **GL Compatibility** | 古風2Dに十分で、低スペック端末・Web も含め最も広く動く |

## アーキテクチャの中心思想：データとコードの分離

- **シナリオ（データ）** = `scenario/*.md`。世界観・マップ・NPC・会話・クエスト・アイテムを記述。AIで育てる対象。
- **エンジン（コード）** = `src/`。markdown を直接は解釈せず、変換済みの `data/*.json` を読む。
- **変換ツール** = `tools/`。ビルド時に markdown → JSON へ変換・検証する。

この分離により:
- シナリオは AI で育て、エンジンはアジャイルに育てる、という両輪が独立して回せる
- 実行時に markdown パーサや AI を持ち込まずに済み、ランタイムが軽く・再現性が高い
- 将来 `src/core/` を別作品でも使える汎用エンジンとして抽出しやすい

## src/core と src/game の分離

最初から2つに分ける:
- `src/core/` … マップ描画・移動・会話・データ読み込みなど、どんなRPGでも使う汎用機構
- `src/game/` … この作品固有のシーン構成・調整・固有ロジック

「1本作ってからエンジンを抽出する」方針なので、最初は境界が曖昧でよい。固有部分が育つにつれ、汎用部分を `core/` に寄せていく。

## AIによるシナリオ育成（開発時）

想定ワークフロー:
1. 作者が世界観やイベントの「種」を短く書く
2. Claude Code のスキルが、その種を読み、`scenario/` の記法に沿って展開・補完・検証する
3. 人がレビューして markdown を確定させる
4. `tools/` が JSON へ変換し、Godot が読み込む

スキルは `.claude/skills/` に置く（S4 で着手予定）。

## 非目標（現時点）

- 実行時のAI動的生成（将来の実験対象。まずは開発時育成に集中）
- 3D・高度なシェーダ表現
- マルチプレイ
